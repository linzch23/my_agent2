# 非编码长任务完整测试方案：城市音乐节应急保障

## 项目启动
在 PowerShell 里运行：

```powershell

cd "D:\agent架构比赛\my_agent2"
$env:PYTHONPATH="src"
$env:MY_AGENT_WEB_PORT="8766"
.\.venv\Scripts\python.exe -m my_agent2.server

```

## 测试目标

这个用例用于同时验证：

- 长任务多轮对话能在左侧对话面板稳定展示。
- 会话树在没有 fork 时保持主干同级缩进。
- 从历史节点 fork 后，树视图能用缩进表达兄弟分支。
- `remember` 能写入长期记忆。
- `search_context`、`read_context`、`show_context_links` 能展示 ContextFS（上下文文件系统）、L0/L1/L2 和 MemoryGraph（记忆图谱）效果。
- `/compact` 后能触发会话压缩和记忆提交，并在后续对话中召回相关上下文。

## 准备步骤

1. 清空 `memory/` 后启动 Web 服务。
2. 打开前端：`http://127.0.0.1:8766/`。
3. 点击左上角 **新会话**。
4. 新会话标题填写：`城市音乐节应急保障测试`。
5. 确认顶部显示新的 `session: <id>`。

## 主干对话脚本

### 第 1 轮：建立项目背景

用户输入：

```text
我们要为“海岚市国庆 3 天大型音乐节”制定一份应急保障方案。请先只做背景理解和关键假设，不要直接写完整方案。已知条件：预计每日 4 万人，场地在滨海新区露天公园，周边有两条主干道、一个地铁站、两个临时停车场；城市常见风险包括台风尾流、短时强降雨、交通拥堵、人员中暑和网络舆情。请输出不超过 8 条关键假设，并说明哪些假设后续需要验证。
```

预期：

- 左侧出现用户输入和 Agent 分析。
- 这轮不要求写长期记忆。
- 树视图中新增一条主干节点，不应逐层缩进。

### 第 2 轮：写入长期项目约束

用户输入：

```text
请记住这个项目的长期约束：海岚市音乐节保障方案必须优先考虑公共安全、极端天气和交通疏导；方案不能依赖新增外部数据库或复杂 SaaS 平台，只能使用本地文件、人工值守表格、现有城市热线和部门微信群。请用 remember 写入长期记忆，类别用 constraints，标题用“海岚市音乐节保障边界”。
```

预期工具：

- `remember`

右侧工具可视化预期：

- 点击左侧 `remember` 工具调用。
- 右侧应显示 **记忆写入路径**。
- 重点能看到：
  - URI 类似 `mem://project/constraints/...`
  - `index.jsonl`
  - `mem/.../*.md`
  - `diffs.jsonl`
  - `links.jsonl`

### 第 3 轮：写入用户偏好

用户输入：

```text
再记住一个用户偏好：我希望这类应急方案输出时先给“指挥长能听懂的一页纸摘要”，再给详细执行表；不要一上来堆概念。请用 remember 写入长期记忆，类别用 preferences，标题用“应急方案输出偏好”。
```

预期工具：

- `remember`

预期：

- 记忆类别应进入 `preferences`。
- 后续回答应该优先采用“一页纸摘要 + 详细执行表”的结构。

### 第 4 轮：生成完整初稿

用户输入：

```text
现在基于刚才的背景和你记住的偏好，生成第一版完整应急保障方案。结构必须是：1）一页纸摘要；2）四阶段执行表：活动前 7 天、活动前 24 小时、活动期间、活动后 48 小时；3）指挥中心信息流转表；4）五个最高风险点；5）哪些内容应该进入长期记忆，哪些只保留在本次会话。
```

预期：

- Agent 应自然采用第 3 轮的偏好。
- 如果模型没有自动召回，可继续第 5 轮显式测试上下文检索。

### 第 5 轮：显式测试 ContextFS 检索

用户输入：

```text
用 search_context 搜索“海岚市 音乐节 保障 边界”，然后告诉我你检索到了哪些长期约束。不要直接改方案。
```

预期工具：

- `search_context`

右侧工具可视化预期：

- 点击 `search_context`。
- 右侧应显示 **ContextFS 检索流程**。
- 重点能看到：
  - Query → Search L0/L1 → Graph Walk → Select Text
  - 检索结果列表
  - L0 / L1 / L2 分层说明

### 第 6 轮：显式测试 L0/L1/L2 读取

用户输入：

```text
请用 read_context 读取刚才命中的“海岚市音乐节保障边界”记忆。先读取 auto 层，再说明如果读取 full 层会多看到什么。
```

预期工具：

- `read_context`

右侧工具可视化预期：

- 点击 `read_context`。
- 右侧应显示 **L0 / L1 / L2 读取策略**。
- 重点验证：
  - `auto` 默认对应 L1 overview。
  - `full` 对应 L2 完整正文。
  - 右侧展示存储路径和选中文本，而不是在左侧展开大量工具结果。

### 第 7 轮：测试 MemoryGraph 链接

用户输入：

```text
请用 show_context_links 查看“海岚市音乐节保障边界”这条记忆的链接关系。如果没有链接，请解释为什么；如果有链接，请说明它如何帮助后续方案召回。
```

预期工具：

- `show_context_links`

右侧工具可视化预期：

- 点击 `show_context_links`。
- 右侧应显示 **MemoryGraph 节点图**。
- 重点验证：
  - seed 节点
  - derived_from / related / supports 等关系边
  - 邻近节点游走路径

## Fork 操作测试

### Fork 位置

切换到左侧 **树状结构**，找到第 4 轮 Agent 输出“完整初稿”的节点。

操作：

1. 点击该 Agent 输出节点。
2. 点击顶部操作条中的 **Fork**。
3. 切回 **对话**。

### Fork 分支 A：医疗急救强化

用户输入：

```text
从刚才完整初稿这个节点分叉。现在只深化“医疗急救和中暑处理”这一条，不要改其他模块。请给出现场医疗点位配置、急救转运路径、志愿者识别高风险人群的方法，以及活动期间每 2 小时的巡查节奏。
```

预期：

- 树视图中应出现从第 4 轮节点派生出的新分支。
- 分支节点相对主干缩进一级。
- 后续分支内如果没有再次 fork，应保持同一缩进，不继续逐层右移。

### Fork 分支 B：舆情响应强化

操作：

1. 回到树状结构。
2. 再次点击第 4 轮“完整初稿”节点。
3. 点击 **Fork**。
4. 切回对话。

用户输入：

```text
从同一个完整初稿节点再开一条分支。这次只深化“网络舆情响应”，不要改医疗和交通模块。请设计舆情监测关键词、分级响应标准、回应口径模板、谣言澄清流程，以及什么时候必须升级到市级指挥中心。
```

预期：

- 树视图中第 4 轮节点下面应出现两个兄弟分支。
- 两个分支应在同一缩进层级。
- 分支 B 不应覆盖分支 A。

## Jump 操作测试

操作：

1. 切到树状结构。
2. 点击第 5 轮 `search_context` 之后的 Agent 解释节点。
3. 点击 **Jump**。
4. 切回对话。

用户输入：

```text
基于你刚才检索到的长期约束，只重新写“一页纸摘要”，不要展开完整方案。
```

预期：

- 当前 active leaf 应跳回第 5 轮附近。
- 新回复应只生成摘要。
- 树视图中应体现 jump 后的新延续路径。

## Label 操作测试

操作：

1. 切到树状结构。
2. 选择第 4 轮“完整初稿”节点。
3. 点击 **Label**。
4. 输入标签：`baseline-plan`。

预期：

- 节点标题或标签区域显示 `baseline-plan`。
- 后续评审时可以快速定位基线方案节点。

## Clone 操作测试

操作：

1. 保持当前 active branch。
2. 点击树操作条中的 **Clone**。

预期：

- 前端顶部 `session: <id>` 更新为新 session。
- 克隆后的 session 保留当前 active branch 内容。
- 适合把某条分支单独变成新的评审会话。

## 压缩与记忆提交测试

用户输入：

```text
请对当前会话进行压缩整理。压缩后请总结：哪些内容应该进入长期记忆，哪些只应作为本次会话归档。然后执行 /compact 或等价的压缩提交流程。
```

如果 CLI 命令无法通过自然语言触发，则在 CLI 或可用接口中手动执行：

```text
/compact
```

预期：

- 会话树出现 `compaction` 节点。
- `SessionMemoryCommitter（会话记忆提交器）` 应把会话归档写入 ContextFS。
- `memory/context/index.jsonl` 应出现新的 `ctx://sessions/archives/...` 记录。
- 如果提取出了长期记忆，应出现新的 `mem://...` 记录。

## 最终召回测试

点击 **新会话**，标题填写：

```text
音乐节方案二次评审
```

新会话中输入：

```text
我们继续讨论海岚市音乐节应急保障。请先用 search_context 搜索“海岚市 音乐节 应急 保障”，再基于长期记忆告诉我：这个项目有哪些必须遵守的约束和输出偏好？
```

预期工具：

- `search_context`
- 可选 `read_context`
- 可选 `show_context_links`

预期结果：

- 应召回第 2 轮的项目边界。
- 应召回第 3 轮的输出偏好。
- 如果已执行压缩提交，应能召回 session archive。
- 右侧可视化能展示检索路径、L0/L1/L2 选择和 MemoryGraph 关系。

## 前端验收清单

| 验收点 | 通过标准 |
|---|---|
| 新会话 | 点击 **新会话** 后顶部 `session: <id>` 更新 |
| 长对话 | 左侧能承载多轮长文本，不把工具大结果直接展开 |
| 工具可视化 | 点击 `remember/search_context/read_context/show_context_links` 后右侧显示对应可视化 |
| 树主干 | 没有 fork 的链状节点保持同级缩进 |
| Fork | 从同一历史节点 fork 两次后出现兄弟分支缩进 |
| Jump | jump 后新消息接在选中历史节点之后 |
| Label | 节点能打标签并用于定位 |
| Clone | 当前 active branch 能克隆成新 session |
| 压缩 | `/compact` 后树里出现 compaction 节点 |
| 记忆召回 | 新 session 能通过 search_context 找回旧会话沉淀的约束和偏好 |

## 注意事项

- `search_context`、`read_context`、`show_context_links`、`remember` 是 Agent 工具，不是独立 HTTP API。
- 前端右侧可视化应从 `runs.toolCalls` 或 `toolEvents` 驱动。
- 清空 `memory/` 不等于清空 `sessions/`；如果要完全从零测试，也可以另行清空 `sessions/`，但会失去现有 fork 演示数据。
